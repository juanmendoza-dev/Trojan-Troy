**Goal:** Replace the flat one-key-per-session model with a Double Ratchet (fresh key per message → forward secrecy + post-compromise "self-healing"), move channel/id/mimeType inside the ciphertext, and pad payloads to size buckets. Zero UX change — bytes on the wire only. Closes review findings H4, M2, M6, L4, H2, and partially B12.

**Architecture:** New pure-logic crypto modules (`aead`, `kdf`, `framing`, `ratchet`) composed by a `ratchetSession` protocol module, wired into `App.tsx`'s send/receive. The post-handshake content/signal envelopes collapse into one opaque `msg` type — the relay already forwards unknown types verbatim, so **no server change is required for the ratchet**. Relay hardening (review H3/M1/M5/L5) is a genuinely independent Track B on its own branch (§Parallel Track B).

**Tech Stack:** TypeScript, React, Vite, Vitest, `libsodium-wrappers` ^0.7.15 — **no new dependencies.** Verified present in the shipped (non-sumo) build: `crypto_scalarmult`, `crypto_aead_xchacha20poly1305_ietf_encrypt/decrypt`, `crypto_generichash` (keyed BLAKE2b), `crypto_kx_keypair`.

## Global Constraints

- Never implement custom cryptographic primitives — libsodium only, no hand-rolled crypto. Every construction here is a composition of the audited primitives above (`AGENTS.md`, `roadmap.md`, review §1).
- The relay only ever routes opaque JSON envelopes — never parses/inspects payloads beyond the `type` field needed to route. No server change in the ratchet track (`decisions.md`).
- Session crypto is ephemeral — ratchet state is held in memory per session, never persisted. A refresh loses the conversation; no reconnect (`decisions.md`, Local Profiles spec keeps this).
- **Preserve the review §2 invariants:** `crypto_kx` rx/tx separation stays load-bearing (it seeds the root key + directional subkeys); the safety number stays 160-bit BLAKE2b over the two sorted **handshake** pubkeys, unchanged; the relay stays blind.
- **Roadmap reordering must be logged before building.** This builds 5.2 ahead of remaining 5.1 work and drops 5.2's old dependency on persistent identity. Per `AGENTS.md`, update `roadmap.md` + add a `decisions.md` entry **before Task 1** (Task 0 below).
- Commit messages: short, plain-language, human-sounding — no AI verbosity, no extra trailers, never AI-authored or co-authored (`AGENTS.md`).
- **Run `git commit` from PowerShell, not the Bash tool** — Git-Bash's bundled `gpg` reads a different keyring and signing silently fails there; native Windows `gpg` (PowerShell) signs correctly. `git add`/`git push` are fine from either shell. A `post-commit` hook auto-pushes every commit — no manual push step.
- Commit early and often — one commit per task minimum (`AGENTS.md`).
- Pure-logic modules get Vitest coverage; screens/`App.tsx` are verified by a throwaway protocol script + manual two-browser pass (no browser automation here) — the standing convention (`decisions.md` 2026-07-20).

### Task 2: Key-derivation module (`crypto/kdf.ts`)

**Files:** Create `client/src/crypto/kdf.ts`, `client/src/crypto/kdf.test.ts`.

**Interfaces:**
- `deriveRootKey(rx, tx): Promise<Uint8Array>` — `RK₀`; keyed BLAKE2b over the **byte-sorted** pair `{rx,tx}` with context `"TTr:root:v2"`, 32 bytes. (Sorted so both roles compute the same value.)
- `kdfRoot(rk, dh): Promise<{ rk, ck }>` — `generichash(64, "TTr:rk:v2"‖dh, key=rk)` split 32/32.
- `kdfChain(ck): Promise<{ ck, mk }>` — `mk = generichash(32, [0x01], key=ck)`, `ck' = generichash(32, [0x02], key=ck)`.
- `deriveChannelSubkey(dirKey, channel): Promise<Uint8Array>` — `generichash(32, "TTr:sub:"+channel, key=dirKey)`; caller passes `tx` for send-subkeys, `rx` for receive-subkeys.

- [ ] **Step 1: Write failing tests** — `deriveRootKey(rx,tx) === deriveRootKey(tx,rx)` (sort-invariant) and equals the value computed from the swapped `(rx,tx)` a responder would hold; `kdfRoot`/`kdfChain` are deterministic and domain-separated (different context/tag → different output); `kdfChain` advances (`ck' !== ck`, `mk !== ck`); `deriveChannelSubkey` differs per channel and per direction key.
- [ ] **Step 2–4: fail → implement → pass.** Implementation is thin wrappers over `sodium.crypto_generichash`.
- [ ] **Step 5: Commit** (PowerShell): `git commit -m "Add ratchet key-derivation module"`

### Task 3: Framing + padding (`crypto/framing.ts`)

**Files:** Create `client/src/crypto/framing.ts`, `client/src/crypto/framing.test.ts`.

**Interfaces:**
- `type Channel = "text" | "voice" | "presence" | "ack" | "profile"`
- `interface Frame { channel: Channel; id: string; mimeType?: string; kind?: "delivered" | "read"; body: Uint8Array }`
- `frame(f: Frame): Uint8Array` — build `uint16LE(len(metaJSON)) ‖ metaJSON ‖ body`, then pad: `uint32LE(len(inner)) ‖ inner ‖ zeros(bucket − 4 − len(inner))`, bucket = smallest of the schedule ≥ `len(inner)+4`.
- `unframe(bytes: Uint8Array): Frame` — read the `uint32LE` length, slice `inner`, parse meta, return raw `body`.
- Schedules (exported consts, tunable): `SCHEDULE_CONTENT = [256, 1024, 4096, 16384]` then round up to the next 16 KiB; `SCHEDULE_SMALL = [64, 256]` for presence/ack/profile.

- [ ] **Step 1: Write failing tests** — `frame → unframe` round-trips for text (body = UTF-8), voice (large binary body + `mimeType`), and ack (`kind`, empty body); the padded length equals the expected bucket; a payload 1 byte over a bucket lands in the next bucket; `body` stays raw (no base64 inflation for a 1 MB voice body → padded length ≈ next 16 KiB multiple, not ~1.33×); malformed input throws (not crashes).
- [ ] **Step 2–4: fail → implement → pass.** Pure byte manipulation; `metaJSON` via `JSON.stringify` + `sodium.from_string`/`to_string` (or `TextEncoder`).

### Task 4: Ratchet core (`crypto/ratchet.ts`) — the heart

**Files:** Create `client/src/crypto/ratchet.ts`, `client/src/crypto/ratchet.test.ts`.

**Interfaces:**
- `interface KeyPair { publicKey: Uint8Array; privateKey: Uint8Array }`
- `interface RatchetHeader { dh: string; pn: number; n: number }`
- `interface RatchetState { DHs; DHr; RK; CKs; CKr; Ns; Nr; PN; MKSKIPPED }` (see spec).
- `initAlice(rk0, bobInitialPub): Promise<RatchetState>` / `initBob(rk0, bobHandshakeKeypair): Promise<RatchetState>`
- `ratchetEncrypt(state, plaintext): Promise<{ header: RatchetHeader; payload: string }>` (mutates `state`)
- `ratchetDecrypt(state, header, payload): Promise<Uint8Array>` (mutates `state`; throws → caller drops)
- Consts: `MAX_SKIP = 100` (per chain), `MAX_SKIPPED_TOTAL = 1000` (global cache cap).

**Implement strictly to the spec's pseudocode** (`initAlice/initBob`, `ratchetEncrypt`, `ratchetDecrypt`, `skipMessageKeys`, `dhRatchet`, `KDF_RK`/`KDF_CK` via Task 2, DH via `crypto_scalarmult`, keypairs via `crypto_kx_keypair`, encryption via Task 1 with `aad = from_string(JSON(header))`). Wipe each `mk` with `sodium.memzero` after use.

- [ ] **Step 1: Write the failing test suite** — this task is defined by its tests. Set up a helper that inits an Alice state and a Bob state sharing an `rk0` (from Task 2 over a real `crypto_kx` exchange) and Bob's initial keypair, then exercise:
  - in-order round-trip both directions;
  - a long alternating conversation (≥ 20 turns, forcing many DH steps) stays in sync;
  - **out-of-order within a chain:** deliver msg 3 before msg 2 → both decrypt correctly (skipped-key path);
  - **out-of-order across a DH step** (deliver a message from the previous chain after the ratchet advanced) → decrypts via `pn` skip;
  - **replay dropped:** re-deliver an already-decrypted message → `ratchetDecrypt` throws, no state corruption, the next real message still decrypts;
  - **skipped-key cap:** a header with `n` beyond `MAX_SKIP` → throws (no unbounded loop); global cache never exceeds `MAX_SKIPPED_TOTAL`;
  - **header tamper:** mutate `header.n`/`header.dh` → AEAD tag fails → throws (AAD binding);
  - **reflection:** feed Alice her own `{header,payload}` → throws (won't open under her receiving chain);
  - `initAlice`/`initBob` produce a working first message from Alice→Bob (Bob's first receive triggers his first DH step).
- [ ] **Step 2: Run tests, verify fail.**
- [ ] **Step 3: Implement to the spec pseudocode.**
- [ ] **Step 4: Run tests, verify pass.** This is the correctness gate for the whole phase — do not proceed until every case above is green.
