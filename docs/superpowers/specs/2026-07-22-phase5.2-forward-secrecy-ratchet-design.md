# Phase 5.2 — Forward-Secrecy Ratchet (Double Ratchet): Design Spec

Status: Draft (brainstormed with Jay 2026-07-22; awaiting approval)
Date: 2026-07-22

## Purpose

Today every message in a session is sealed under **one static session key**
(`crypto_secretbox_easy` with a fresh nonce, key from the one-shot `crypto_kx`
handshake). That is correct but flat: if that session key ever leaks — a grabbed
tab, an XSS bug, a memory dump — **every message in the conversation, past and
future, is readable.** There is no forward secrecy and no recovery from a
transient compromise.

This phase replaces the flat session key with a **Double Ratchet** — the
per-message key-rotation protocol Signal/WhatsApp/iMessage use — built entirely
from primitives already in the shipped `libsodium-wrappers` build. It gives:

- **Forward secrecy** — each message gets its own key, deleted right after use;
  stealing today's state does not unlock yesterday's messages.
- **Post-compromise security ("self-healing")** — after a transient compromise,
  the next DH ratchet step re-establishes secrecy the attacker can't follow.
- **Intrinsic replay/reorder protection** — the per-message chain counter kills
  replays and detects reordering for free.

Two smaller under-the-hood upgrades ride the same refactor:

- **Sealed framing** — the logical channel (text/voice/ack/…), the `messageId`,
  and voice `mimeType` move *inside* the encrypted payload. The relay stops
  seeing them.
- **Size-bucket padding** — every payload is padded to a bucket before
  encryption, so the relay can't read exact message lengths.

**This is invisible to the user.** No screen, flow, or interaction changes —
only the bytes on the wire. Per Jay's Phase 5 steer (`decisions.md`, and the
`phase-5-security-direction` memory), all UX stays exactly as it ships today.

## What this closes (from the Phase 4.7 review)

- **H4** (no replay/reorder/drop protection) — the ratchet's chain counters
  subsume it. Dedup is on the **ratchet counter `n` + ratchet public key**, never
  a cleartext id (the review's critical detail).
- **M2** (unauthenticated envelope `type` → cross-channel confusion) — the true
  channel moves inside the ciphertext and each static channel uses a distinct
  subkey; a relabel fails to decrypt.
- **M6** (forgeable/oracle cleartext receipts) — receipts are sealed and become
  unforgeable, and are no longer a decrypt-success oracle (indistinguishable
  from any other `msg` on the wire).
- **L4** (cleartext `messageId` correlation) — the id moves inside the payload.
- Partial **B12** (traffic analysis) — padding hides sizes; documented residuals
  remain (see §Residuals).

## Invariants preserved (must not regress — see review §2)

- **`crypto_kx` rx/tx direction separation stays load-bearing.** It still seeds
  the ratchet root and the per-channel subkeys; a reflected ciphertext still
  won't open (see §Reflection resistance).
- **Safety number is unchanged** — still 160-bit BLAKE2b over the two sorted
  *handshake* public keys (`safetyNumber.ts`). The ratchet's ephemeral keys
  rotate and are deliberately **not** part of the safety number. (Re-binding the
  number to the session — review L2 — is explicitly out of scope here.)
- **The relay stays architecturally blind.** It forwards opaque `msg` envelopes;
  it never gains the ability to read plaintext, channel, id, or exact size.
- **Audited-library-only.** Every construction below is a composition of
  libsodium primitives (`crypto_scalarmult`, `crypto_generichash`,
  `crypto_aead_xchacha20poly1305_ietf`). No hand-rolled primitive — this is the
  same category as today's `crypto_kx` + `crypto_secretbox` composition.

## Hard constraints (carried, all satisfied)

- Never hand-roll primitives — audited libraries only. ✅
- Relay never reads plaintext. ✅
- No live/streaming calling or P2P. ✅ (unchanged; this is async messaging)

## Cryptographic design

### Primitives (all verified present in `libsodium-wrappers` ^0.7.15)

| Role | Primitive |
|---|---|
| DH (X25519) for the ratchet step | `crypto_scalarmult(priv, pub)` |
| Root KDF & chain KDF | `crypto_generichash(len, msg, key)` (keyed BLAKE2b) |
| Message / channel encryption | `crypto_aead_xchacha20poly1305_ietf_encrypt/decrypt` (AAD-capable) |
| Ratchet keypairs | `crypto_kx_keypair()` (raw X25519 keys, usable with `crypto_scalarmult`) |

> `crypto_kdf_derive_from_key` is also present and is an equally valid choice for
> the chain KDF, but we standardize on keyed `crypto_generichash` because it's
> already used (`safetyNumber.ts`), is guaranteed in this build, and expresses
> both "keyed hash over input" (root KDF) and "derive two subkeys" (chain KDF)
> uniformly. No `-sumo`, no bundle-size change.

### Bootstrapping the root key from the existing handshake

The handshake is **unchanged** (pubkey exchange, `deriveSessionKeys`, safety
number). The ratchet consumes its output:

- Both sides already hold `{ rx, tx }` from `crypto_kx`, where
  initiator.tx == responder.rx and initiator.rx == responder.tx. The unordered
  pair `{rx, tx}` is therefore identical on both sides.
- **Initial root key:** `RK₀ = generichash(32, msg = sortBytes(rx, tx), key = "TTr:root:v2")`.
  Both sides compute the same value; no new DH, reuses trusted key material.
- **Per-channel static subkeys** (for the non-ratcheted channels, see below) are
  derived *directionally* to keep reflection resistance:
  `Kch_tx = generichash(32, "TTr:sub:"+ch, key = tx)`,
  `Kch_rx = generichash(32, "TTr:sub:"+ch, key = rx)`.
  Encrypt outgoing with `Kch_tx`, decrypt incoming with `Kch_rx`.

### Two channel classes (a deliberate split)

Not everything needs per-message ratcheting. Conversation **content** does;
ephemeral non-content signals don't and would pollute the ratchet.

- **Ratcheted (full forward secrecy + PCS): `content`** — text and voice
  messages. These carry a ratchet header and advance the Double Ratchet.
- **Static per-channel subkey (sealed + unforgeable, not per-message-ratcheted):
  `presence`, `ack`, `profile`.** Sealed under the directional subkeys above.
  Rationale: presence is a ~2.5s heartbeat — ratcheting it would spin the chain
  counter and inflate skipped-key bookkeeping for zero confidentiality gain;
  acks/profile carry no message content. They still gain unforgeability (M6) and
  channel separation (M2) from the distinct subkeys + AEAD.

> *Alternative considered:* ratchet **everything** (one path, simplest mental
> model, best metadata uniformity) at the cost of heartbeat-driven chain churn
> and heavier skipped-key handling. Recommended design is the split; this is an
> easy knob to flip if the split proves fiddly in implementation.

### Ratchet state (in memory only, per session)

```ts
interface RatchetState {
  DHs: KeyPair;                       // our current ratchet keypair
  DHr: Uint8Array | null;             // peer's current ratchet public key
  RK: Uint8Array;                     // root key (32)
  CKs: Uint8Array | null;             // sending chain key
  CKr: Uint8Array | null;             // receiving chain key
  Ns: number; Nr: number; PN: number; // send count, recv count, prev-chain length
  MKSKIPPED: Map<string, Uint8Array>; // "dhrPubB64:n" -> message key (bounded)
}
```

No at-rest persistence — the ratchet is ephemeral like today's session keys
(Layer-B history, if built, saves the *decrypted transcript*, never ratchet
state). Zeroize (`sodium.memzero`) `RK/CK*/MK/DHs.priv` on leave (ties review
L3/B13).

### Init (maps to the existing `role`)

Initiator = "Alice", responder = "Bob". Bob's *initial* ratchet keypair is his
existing handshake `crypto_kx` keypair; Alice uses Bob's handshake public key as
her initial `DHr`. (Reusing Bob's handshake keypair for one ratchet DH avoids any
handshake wire change; a dedicated ratchet prekey is a clean future upgrade if we
ever want to retire the reuse — noted, not needed now.)

```
KDF_RK(RK, dh):                        // dh = 32-byte X25519 output
  okm = generichash(64, "TTr:rk:v2" ‖ dh, key=RK)
  return (okm[0:32] as RK', okm[32:64] as CK)

KDF_CK(CK):
  MK  = generichash(32, [0x01], key=CK)
  CK' = generichash(32, [0x02], key=CK)
  return (CK', MK)

DH(priv, pub) = crypto_scalarmult(priv, pub)

initAlice(RK0, bobInitialPub):
  DHs = crypto_kx_keypair();  DHr = bobInitialPub
  (RK, CKs) = KDF_RK(RK0, DH(DHs.priv, DHr))
  CKr=null; Ns=Nr=PN=0; MKSKIPPED={}

initBob(RK0, bobHandshakeKeypair):
  DHs = bobHandshakeKeypair;  DHr = null
  RK = RK0; CKs=null; CKr=null; Ns=Nr=PN=0; MKSKIPPED={}
```

### Send / receive

```
ratchetEncrypt(state, plaintext):       // plaintext = framed+padded inner bytes
  (state.CKs, MK) = KDF_CK(state.CKs)
  header = { dh: b64(state.DHs.publicKey), pn: state.PN, n: state.Ns }
  state.Ns += 1
  nonce = randombytes(24)
  ct = aead_encrypt(plaintext, aad=JSON(header), nonce, MK)   // header authenticated
  memzero(MK)
  return { header, payload: b64(nonce ‖ ct) }

ratchetDecrypt(state, header, payload):
  k = header.dh + ":" + header.n
  if MKSKIPPED.has(k):                                   // out-of-order arrival
    MK = MKSKIPPED.get(k); MKSKIPPED.delete(k)           // consume → replay-safe
    return aead_decrypt(payload, aad=JSON(header), MK)   // AEAD verifies header
  if b64(state.DHr) !== header.dh:                        // new ratchet pub → DH step
    skipMessageKeys(state, header.pn)
    dhRatchet(state, header)
  else if header.n < state.Nr:                            // already consumed in this chain
    throw "replay/too-old"                                // MK already deleted → drop
  skipMessageKeys(state, header.n)
  (state.CKr, MK) = KDF_CK(state.CKr); state.Nr += 1
  return aead_decrypt(payload, aad=JSON(header), MK)      // throw on tag mismatch → dropped

skipMessageKeys(state, until):
  if state.CKr == null: return
  if state.Nr + MAX_SKIP < until: throw "too many skipped"   // DoS bound
  while state.Nr < until:
    (state.CKr, MK) = KDF_CK(state.CKr)
    MKSKIPPED.set(b64(state.DHr)+":"+state.Nr, MK)           // bounded (see caps)
    state.Nr += 1

dhRatchet(state, header):
  state.PN = state.Ns; state.Ns = 0; state.Nr = 0
  state.DHr = fromB64(header.dh)
  (state.RK, state.CKr) = KDF_RK(state.RK, DH(state.DHs.priv, state.DHr))
  state.DHs = crypto_kx_keypair()
  (state.RK, state.CKs) = KDF_RK(state.RK, DH(state.DHs.priv, state.DHr))
```

- **Replay (H4):** a replayed message is either an already-consumed `n` (MK
  deleted → `n < Nr` guard drops it) or, if still in `MKSKIPPED`, decrypts once
  and is then removed (second replay drops). Dedup is on `(dh, n)`, never a
  cleartext id.
- **Header integrity:** `header` is the AEAD **associated data**, so tampering
  the DH pub / counters fails the tag → dropped (no silent re-key).
- **Caps:** `MAX_SKIP = 100` per chain; global `MKSKIPPED` cap `= 1000` (evict
  oldest / refuse). Bounds memory against a relay minting a huge `n`.

### Reflection resistance (preserves review §2)

A relay reflecting your own `content` ciphertext back: its `header.dh` is *your*
current ratchet pub, so you'd attempt a DH step against your own key and derive
keys that don't match the sealing keys → AEAD tag fails → dropped. Static
channels use directional subkeys (`Kch_tx` ≠ `Kch_rx`), so a reflected
presence/ack won't open under your `rx` subkey. Both hold the line the current
rx/tx split holds today.

## Framing & padding (`crypto/framing.ts`, pure)

The inner plaintext (before ratchet/AEAD) is length-prefixed metadata + raw body,
then padded to a bucket. Keeping the body raw (not base64-in-JSON) avoids
inflating voice clips.

```
inner  = uint16LE(len(metaJSON)) ‖ metaJSON ‖ bodyBytes
meta   = { v:2, ch:"text"|"voice"|"presence"|"ack"|"profile", id, mt? }
         // text: body = UTF-8 text; voice: body = clip bytes, mt = mimeType;
         // ack: meta = { ..., kind:"delivered"|"read", id }; body empty
padded = uint32LE(len(inner)) ‖ inner ‖ zeros(bucket - 4 - len(inner))
bucket = smallest B in SCHEDULE with B >= len(inner)+4
```

- `SCHEDULE_CONTENT = [256, 1024, 4096, 16384]`, then round up to the next
  16 KiB for larger voice. `SCHEDULE_SMALL = [64, 256]` for presence/ack/profile.
  (Values tunable; confirm the largest legit voice clip against the relay
  `maxPayload` chosen in Track B.)
- Unpad: read `uint32LE`, slice `inner`, parse `meta`, take `body`.
- Padding lives **inside** the AEAD, so it needs no cryptographic property —
  just deterministic bucketing. `id` and `mt` are now confidential (L4, + the
  latent voice-`mimeType` sink from review L6).

## Wire format (`net/relayClient.ts`)

Collapse the post-handshake content/signal envelopes into **one opaque type** so
the relay can't distinguish them. Handshake envelopes are unchanged.

```ts
export type Envelope =
  | { type: "create" }
  | { type: "created"; roomCode: string }
  | { type: "join"; roomCode: string }
  | { type: "peer-connected" }
  | { type: "peer-disconnected" }
  | { type: "pubkey"; payload: string; v: number }        // + protocol version
  | { type: "msg"; c: 0|1|2|3; header?: RatchetHeader; payload: string }
  | { type: "error"; message: string };
// c: 0=content (ratcheted, header present), 1=presence, 2=ack, 3=profile (static subkey)
// RatchetHeader = { dh: string; pn: number; n: number }
```

- The old `ciphertext` / `voice` / `presence` / `profile` / `delivered` / `read`
  envelopes are **removed** and replaced by `msg`. The relay already forwards
  unknown types opaquely, so **no server code change is required** for this
  (Track B is separate hardening).
- `c` is a minimal cleartext key-class selector. It is **not** a confidentiality
  leak beyond today: the relay already sees channel class via `type`. Text-vs-
  voice, id, mimeType, ack-kind, and exact size are now all hidden. Relabeling
  across classes fails (distinct keys / AAD).
- **Protocol version:** add `v` to `pubkey`. On handshake, if the peer's `v`
  differs from `PROTOCOL_VERSION`, route to the error screen ("update needed")
  rather than derive keys — cheap version negotiation for a single-deploy app.

## Data flow (unchanged for the user)

1. Pair + `crypto_kx` handshake + safety number — **exactly as today**.
2. On keys established, both sides derive `RK₀` + static subkeys and init the
   ratchet. Nothing visible changes.
3. Send: frame → pad → `ratchetEncrypt` → `msg c:0`. Each message rotates the key.
4. Receive: `ratchetDecrypt` (handles out-of-order, drops replays) → route on the
   decrypted channel → render / presence / status, exactly as now.
5. Presence/ack/profile: sealed under static subkeys (`c:1/2/3`).
6. Leave: teardown + zeroize.

## Error handling / edge cases

- **Decrypt failure** (tamper, wrong key, version skew): existing
  `decryption-error` bubble; never crash. Replays/too-old are dropped silently.
- **Skipped-key flood:** `MAX_SKIP` / global cap throw → treat as a dropped
  message, not a crash.
- **Out-of-order across a DH step:** `skipMessageKeys(header.pn)` before the step
  preserves the tail of the previous receiving chain (bounded).
- **Presence under load:** static subkey, so a dropped heartbeat costs nothing
  (auto-expiry already exists, `presenceState.ts`).
- **Version mismatch:** detected at `pubkey` (`v`), surfaced as the error screen.
- **A second `pubkey`** (review H2): still guard it — with the ratchet, a
  post-init `pubkey` is a protocol violation → error screen, never re-seed. (H2
  is a tiny standalone patch; fold it in here since we touch `exchangeKeys`.)

## Residuals (documented, not oversold — matches the app's honest posture)

- The relay still sees **channel class** (`c`) and an approximate **size bucket**
  (so "large ⇒ voice" is still inferable). Full class/size uniformity (padding
  everything to one size, cover traffic) is deliberate future scope (B12).
- **Presence/ack/profile are sealed but not per-message forward-secret** (static
  subkey from the session). They carry no message content; this is an accepted
  trade for not spinning the ratchet on a heartbeat.
- Ratchet state is in-memory only; a live-device compromise while unlocked can
  read current keys (out of the stated threat model — same line as everything
  else). Zeroing on leave is best-effort in JS.
- MITM at first contact is still gated by the **safety number** — this phase does
  **not** change that (identity/TOFU was the flow-changing option Jay set aside).
