# Phase 4.5 — Working Prototype Design

Unifies the Iris Glass design as the app's standard look, adds a settings
modal, and gets the prototype hosted on a persistent shareable link. Builds
on top of Phases 1-4's unchanged crypto/relay/audio layers and the theme
system from Phase 4. See `roadmap.md` for where this sits in the overall
build order and `decisions.md` for the non-obvious calls below.

## 1. Iris Glass orb fix (blocking prerequisite)

The `floatOrb` keyframe already exists in `client/src/styles/keyframes.css`
but no element in the app uses it — this is a real gap from the Phase 4
design handoff, not a cosmetic tweak.

Add two orb elements to `ChatScreen`, rendered only when
`data-theme="iris"`, using the exact values from the original design
handoff (`ui/Trojan Troy Desktop Redesign/design_handoff_trojan_troy/`):

- Orb 1: 340px diameter, `top: 100px; left: 340px`,
  `background: radial-gradient(circle, rgba(143,166,255,0.14), transparent 65%)`,
  `filter: blur(24px)`, `animation: floatOrb 9s ease-in-out infinite`
- Orb 2: 300px diameter, `bottom: 60px; right: 80px`,
  `background: radial-gradient(circle, rgba(196,160,255,0.11), transparent 65%)`,
  `filter: blur(24px)`,
  `animation: floatOrb 12s ease-in-out infinite; animation-delay: 1.4s`
- Both: `border-radius: 50%`, `position: absolute`, `pointer-events: none`,
  sit behind chat content (the message list, sidebar, composer, and title
  bar all need `position: relative` + a z-index above the orbs, or the
  orbs container needs a z-index below 0 within `.chat-screen`, which is
  already `position: fixed`).

Pulse Slate's equivalent gap (`glowPulse` central ambient glow, unused) is
explicitly **not** part of this task — stays backlog per `roadmap.md`,
since Iris Glass is becoming the standard design, not Pulse Slate.

## 2. Iris Glass as default design + loading screen unification

- `ThemeContext.readStoredTheme()`'s fallback (used only when
  `localStorage` has no stored theme — i.e., first-time visitors) changes
  from `"apple"` to `"iris"`.
- The theme switcher stays available with all three options (Apple, Iris
  Glass, Pulse Slate) — this only changes what new users see by default,
  it does not remove Apple/Pulse.
- `LoadingScreen` currently always renders in the Apple light/dark style
  regardless of the selected chat theme (a deliberate Phase 4 deviation —
  see `decisions.md`'s 2026-07-19 entry). That hardcoded Apple styling is
  replaced with hardcoded Iris Glass styling instead: dark gradient
  background (`linear-gradient(160deg, #0D0F18 0%, #101223 100%)`),
  periwinkle accents (`#8FA6FF`), Schibsted Grotesk font, and the same two
  ambient orbs from section 1 (reused, not redefined) — so the loading
  screen and the now-default chat theme share one consistent visual
  language, per the Phase 4.5 goal in `roadmap.md`.
- `resolveLoadingScheme()` in `client/src/theme/theme.ts` becomes dead code
  once the loading screen no longer follows light/dark scheme — remove it
  and its call site in `ThemeContext.tsx`/`App.tsx`, and drop its test
  cases from `theme.test.ts`.
- **Not doing yet:** making the loading screen fully theme-aware (a
  different loading style per Apple/Iris/Pulse selection). Explicitly
  deferred — see the roadmap backlog entry below.

## 3. Settings — floating centered modal

A new `Settings` component, opened via a gear icon added to `TitleBar`,
**visible only on the chat screen** (not on the start/waiting/handshake/
safety-number screens — those stay out of scope per the original Phase 4
scope cut, and room/session info and "leave chat" only make sense once
you're actually in a room).

**Presentation:** centered floating modal with a dimmed backdrop (click
outside or a close button dismisses it), similar to claude.ai's
settings/usage panel — not a sidebar drawer or full-screen takeover.
Styled to match the active theme (frosted glass treatment on Iris Glass
using the same `--bg-card`/`--border`/`backdrop-filter` tokens the rest of
the chat chrome already uses, plain card styling on Apple/Pulse).

**Contents (all four in scope for this phase):**

1. **Theme switcher** — the existing Apple/Iris/Pulse picker, moved here.
   Today's always-on floating `ThemeSwitcher` (rendered globally in
   `main.tsx`, visible on every screen — a Phase 4 dev-convenience
   placement, not a designed feature) is removed; `ThemeSwitcher.tsx`'s
   existing button markup/logic is reused inside the modal, just no
   longer independently mounted.
2. **Room/session info** — room code (already available) and the safety
   number. This requires a small plumbing change: `App.tsx`'s `Screen`
   union's `"chat"` variant currently only carries `roomCode`, dropping
   the `safetyNumber` once you leave the safety-number screen. Add
   `safetyNumber: string` to the `"chat"` variant and pass it through in
   the `onVerified` callback (`SafetyNumberScreen` → `setScreen`).
   Connection status is not separately tracked — reaching the chat screen
   at all implies "connected" (a peer disconnect already redirects the
   whole app to the existing full-screen error state via the
   `peer-disconnected` handler in `exchangeKeys`, unchanged by this
   phase), so this panel can just always read "Connected".
3. **Leave chat** — a button that closes the relay connection
   (`clientRef.current?.close()`), clears in-memory session state
   (`sessionKeysRef.current = null`, `setMessages([])`), revokes any
   voice-message object URLs (reusing the existing cleanup logic in
   `App.tsx`'s unmount effect), and returns to `{ name: "start" }`.
4. **About/security info** — static, non-interactive copy describing the
   E2E model, reusing the loading screen's reassurance language ("The
   relay only ever sees ciphertext. Your keys never leave this device.")
   plus a line noting the safety number verifies this session.

Roadmap backlog entry: come back and brainstorm additional settings scope
beyond these four items — this phase intentionally doesn't try to
enumerate everything a settings panel could eventually hold.

## 4. Hosting

**Split hosting**, since the relay is a stateful WebSocket server with
in-memory room state (`RoomManager`) — that doesn't fit Vercel's
serverless/edge model, so only the client goes there.

- **Client → Vercel.** Vercel project root directory set to `client/`
  (Vite auto-detected, no `vercel.json` needed). Environment variable
  `VITE_RELAY_URL` set to the deployed relay's `wss://` URL — the client
  already reads this at `client/src/App.tsx:16`
  (`import.meta.env.VITE_RELAY_URL ?? "ws://localhost:8080"`), so no code
  change is needed there, only the Vercel dashboard/env config.
- **Relay → Render**, using a free web service. No server code changes
  needed — `server/src/index.ts` already reads `process.env.PORT`, and
  `server/package.json`'s existing `build` (`tsc`) and `start`
  (`node dist/index.js`) scripts are exactly what Render's Node service
  expects. Add a `render.yaml` at the repo root so the service config is
  reproducible/in-repo rather than only living in Render's dashboard.
- No CORS handling needed — the `ws` library doesn't enforce
  same-origin/CORS restrictions on WebSocket upgrade requests the way
  browser `fetch` does.
- Both free tiers are acceptable for a hackathon demo link. Render's free
  tier cold-starts after inactivity (peer has to wait ~30-60s on the
  first connection after idle); this is a known, accepted trade-off, not
  something this phase tries to solve.

## Testing

- Iris orb positioning/animation and the settings modal are visual —
  verify with the existing Playwright-based manual verification pattern
  from Phase 4 (see `progress.md`'s Phase 4 entry), not new unit tests.
- `App.tsx`'s `Screen` type change (`safetyNumber` on `"chat"`) and the
  `handleLeave` state-reset logic are logic-bearing — cover with the
  existing Vitest setup, following the pattern already used for
  `theme.test.ts` / `screenOverride.test.ts`.
- Hosting is manual verification (open the Vercel URL, confirm a real
  room pairs and exchanges messages against the Render-hosted relay) —
  no automated test for this.
